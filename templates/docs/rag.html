What is RAG?

Retrieval-Augmented Generation

RAG enhances large language models by retrieving relevant information from a knowledge base before generating responses. This approach allows the model to:

  • Answer questions based on your specific data
  • Provide sourced information rather than generating answers from its training data
  • Reduce hallucinations by grounding responses in factual information
  • Stay up-to-date with your latest content

Our RAG API provides powerful tools for building AI applications that can answer questions and generate text based on your specific documents and content. By combining the power of vector databases with state-of-the-art language models, you can create intelligent systems that deliver accurate, contextually relevant responses.

How RAG Works

1. Create a Vectorstore

Upload documents (PDF, DOCX, TXT, etc.) or text strings to create a vectorstore. The system processes your content by:

  • Splitting documents into smaller, manageable chunks
  • Creating vector embeddings that capture semantic meaning
  • Building an index for fast similarity search

2. Retrieve Relevant Information

When a query is submitted, the system:

  • Converts the query into a vector embedding
  • Searches the vectorstore to find the most semantically similar chunks
  • Retrieves the top matches to use as context

3. Generate Contextual Response

The retrieved information is provided to a language model, which:

  • Analyzes both the original query and the retrieved context
  • Generates a response that answers the query based specifically on the provided context
  • Delivers accurate, factual information grounded in your data

Document Vectorstore
3 AI Credits

Create vectorstores from document files such as PDFs, Word documents, text files, and spreadsheets. Use this approach when you have existing documents that you want to make searchable and queryable.

Features

  • Support for multiple file types (PDF, DOCX, TXT, CSV, XLSX)
  • Automatic document chunking with configurable size and overlap
  • Metadata preservation from original documents
  • Batch processing of multiple files
  • FAISS vector storage for fast similarity search

API Endpoint

POST /rag/vectorstore/document

Create a vectorstore from document files that you've previously uploaded via the /file endpoint.

Required parameters:

  • file_ids - Array of file IDs to process
  • X-Token - Your API authentication token (header)

Optional parameters:

  • vectorstore_name - Name for the vectorstore (default: auto-generated)
  • chunk_size - Size of text chunks (default: 1000)
  • chunk_overlap - Overlap between chunks (default: 200)

Document Processing Best Practices

For optimal results when creating document vectorstores:

  • Use smaller chunks (500-800 characters) for FAQ-style content where precise answers matter
  • Use larger chunks (1000-1500 characters) for documents where context and connections between ideas are important
  • Set chunk_overlap to approximately 10-20% of your chunk_size
  • Group related documents into separate vectorstores for better organization
  • For complex PDFs with tables and layouts, use the advanced endpoint for better extraction

String Vectorstore
2 AI Credits

Create vectorstores directly from text strings without requiring document uploads. Ideal for API responses, short-form content, or information already available as text.

Features

  • Direct text-to-vectorstore creation
  • Custom metadata support
  • Optimized for smaller text chunks
  • Fast processing with minimal overhead
  • Ideal for dynamic or frequently updated content
  • Lower AI credit cost compared to document processing

API Endpoint

POST /rag/vectorstore/string

Create a vectorstore directly from text content without requiring document upload.

Required parameters:

  • content - Text content to create the vectorstore from
  • X-Token - Your API authentication token (header)

Optional parameters:

  • content_source - Source identifier (default: "User Input")
  • vectorstore_name - Name for the vectorstore (default: auto-generated)
  • chunk_size - Size of text chunks (default: 500)
  • chunk_overlap - Overlap between chunks (default: 100)
  • metadata - Additional metadata to store with the content

When to Use String Vectorstores

String vectorstores offer advantages for certain use cases:

  • Working with content from APIs or databases that's already in text format
  • Creating vectorstores from short, structured information like product details or FAQs
  • Frequently updated content that requires regular refresh
  • When you need more control over metadata and content organization
  • For small pieces of information you want to retrieve precisely

Querying Vectorstores

Once you've created a vectorstore, you can query it to retrieve information and generate responses using our LLM models. This is the core RAG functionality that allows you to get answers based on your specific content.

Query API Endpoint

POST /rag/vectorstore/consume

Required parameters:

  • vectorstore_id - ID of the vectorstore to query
  • query - User question or request to answer
  • X-Token - Your API authentication token (header)

Optional parameters:

  • model - LLM model to use (gpt-4o-mini or gpt-4o, default: gpt-4o-mini)
  • system_prompt - Custom system prompt to guide model behavior
  • temperature - Controls randomness (0=focused, 1=creative, default: 0.5)
  • json_output - Format response as JSON (default: false)
  • include_sources - Include source document references (default: false)

Custom System Prompts

The system_prompt parameter allows you to control how the model uses the retrieved information. You can customize the prompt for different use cases:

Technical Support
"You are a technical support specialist. Use the context to provide step-by-step solutions to technical problems. If the solution isn't in the context, tell the user you don't have that information."
Legal Document Analysis
"You are a legal assistant analyzing contracts. Based on the provided context, explain legal terms in plain language. Do not provide legal advice beyond what's in the context."
Educational Content
"You are a helpful tutor. Explain concepts from the context in simple terms appropriate for a high school student, using analogies when possible."

Query Optimization Tips

For best results when querying vectorstores:

  • Use specific, clearly worded queries rather than vague questions
  • For factual information, use lower temperature settings (0.1-0.3)
  • For creative responses within the document context, use higher temperatures (0.5-0.7)
  • Use the include_sources: true parameter to get source documents in the response (great for debugging)
  • For complex queries, use GPT-4o instead of GPT-4o-mini for more nuanced understanding
  • Craft system prompts that specifically instruct the model on how to interpret and use the retrieved context

Best Practices for RAG Implementation

  • Organize related content - Create separate vectorstores for different domains or topics
  • Optimize chunk sizes - Smaller chunks (300-500 characters) for precise answers, larger chunks (1000-1500 characters) for more context
  • Use descriptive vectorstore names - Makes management and reference easier
  • Customize system prompts - Tailor the LLM behavior to your specific use case
  • Use lower temperatures (0.1-0.3) for factual responses and higher temperatures (0.5-0.7) for more creative applications
  • Clean and preprocess your text - Remove irrelevant content and format consistently for better results

Advanced Options

For more complex document processing needs, our platform offers advanced features to improve the quality of your vectorstores and queries.

Enhanced PDF Processing

The advanced document processing endpoint (/rag/vectorstore/document/advanced) provides improved handling of PDF documents:

  • Better extraction of text from complex layouts
  • Improved handling of tables and structured content
  • Automatic duplicate detection and removal
  • Content cleaning and normalization
  • Batch processing for reliability with large documents
  • Detailed processing statistics

Vectorstore Management

Our platform provides tools to manage your vectorstores efficiently:

  • List all your vectorstores with metadata
  • Load vectorstore details to check document count and settings
  • Delete vectorstores when they're no longer needed
  • Create separate vectorstores for different knowledge domains
  • Track token usage for budgeting and optimization

Git Policy Endpoint

Our platform includes a specialized endpoint for Git policy questions at /rag/vectorstore/consume/git_policies

This pre-configured vectorstore contains comprehensive Git best practices and company policies, providing:

  • Guidance on branch strategies
  • Commit message conventions
  • Pull request workflows
  • Code review standards
  • Merge and release procedures

Unlike other endpoints, this one doesn't require a vectorstore_id parameter, as it uses a pre-indexed knowledge base.
